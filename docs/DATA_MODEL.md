# Data Model

**Product:** SceneFrame.ai  
**Last Updated:** Feb 2026

This document defines every table, its columns, relationships, and the reasoning behind each decision. The goal is to get this right before writing a single migration.

---

## Entity Relationship Overview

```
users
  └── projects (one user → many projects)
        └── scenes (one project → many scenes)
              └── assets are stored as URL columns on scenes (not a separate table)
```

Simple and flat. No over-engineering for MVP.

---

## Tables

---

### `users`

Stores SceneFrame-specific data per user. Clerk is the source of truth for identity — we never store name, email, or password. We only store what Clerk can't hold: the encrypted Replicate API key.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `text` | PRIMARY KEY | Clerk user ID (e.g. `user_2abc...`). Use Clerk's ID directly — no surrogate key needed. |
| `replicate_key_enc` | `text` | nullable | AES-256-GCM encrypted Replicate API key. Null until user completes onboarding. |
| `replicate_key_dek` | `text` | nullable | The Data Encryption Key (DEK), itself encrypted with the env-var KEK. Stored alongside the ciphertext. |
| `replicate_key_iv` | `text` | nullable | Initialisation vector used during encryption. Required for decryption. |
| `onboarding_complete` | `boolean` | NOT NULL, default `false` | True once the user has saved a valid Replicate key. Used to gate access to the dashboard. |
| `created_at` | `timestamp` | NOT NULL, default `now()` | |

**Why three columns for the key?**  
Envelope encryption requires three pieces of data: the encrypted payload (`replicate_key_enc`), the encrypted DEK (`replicate_key_dek`), and the IV (`replicate_key_iv`). All three are needed to decrypt. Splitting them into separate columns is cleaner than a JSON blob and makes it obvious what each piece is for.

---

### `projects`

One project = one creative brief = one Kanban board. A user can have many projects (e.g. one per YouTube video they're producing).

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `uuid` | PRIMARY KEY, default `gen_random_uuid()` | |
| `user_id` | `text` | NOT NULL, FK → `users.id` | Scopes the project to a user. |
| `name` | `text` | NOT NULL | User-defined project name (e.g. "Beach Sunset Promo"). |
| `director_prompt` | `text` | NOT NULL | The original concept the user typed. Preserved for reference and potential re-generation. |
| `status` | `text` | NOT NULL, default `'pending'` | Project-level status: `pending` (script not yet generated), `active` (in progress), `complete` (all scenes done). |
| `created_at` | `timestamp` | NOT NULL, default `now()` | |
| `updated_at` | `timestamp` | NOT NULL, default `now()` | Updated on any change to the project or its scenes. |

**Why store `director_prompt` on the project?**  
Users may want to tweak the original prompt and regenerate the script. Keeping it on the project means we always know what concept the scenes came from.

---

### `scenes`

The core entity. Each scene is a card on the Kanban board. It moves independently through the 4 stages and accumulates asset URLs as it progresses.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `uuid` | PRIMARY KEY, default `gen_random_uuid()` | |
| `project_id` | `uuid` | NOT NULL, FK → `projects.id`, ON DELETE CASCADE | Deleting a project deletes all its scenes. |
| `order` | `integer` | NOT NULL | Scene sequence within the project (1, 2, 3…). Used for display ordering. |
| `title` | `text` | NOT NULL | Short scene title generated by the LLM (e.g. "Aerial coastline flyover"). |
| `description` | `text` | NOT NULL | Full visual description of the scene, used as the prompt for image generation. |
| `stage` | `text` | NOT NULL, default `'script'` | Current Kanban stage: `script`, `images`, `video`, `audio`. |
| `status` | `text` | NOT NULL, default `'idle'` | Job status within the current stage: `idle`, `generating`, `done`, `error`. |
| `error_message` | `text` | nullable | The error from the last failed Trigger.dev job, shown in the UI for the retry flow. |
| `start_image_url` | `text` | nullable | R2 URL of the start frame image. Set when image generation job completes. |
| `end_image_url` | `text` | nullable | R2 URL of the end frame image. Set when image generation job completes. |
| `video_url` | `text` | nullable | R2 URL of the generated `.mp4` clip. Set when video generation job completes. |
| `audio_url` | `text` | nullable | R2 URL of the generated audio file. Set when audio generation job completes. |
| `replicate_prediction_id` | `text` | nullable | The active Replicate prediction ID. Used for logging and debugging. Cleared on job completion. |
| `trigger_job_id` | `text` | nullable | The active Trigger.dev job ID. Used for status lookups if needed. Cleared on job completion. |
| `created_at` | `timestamp` | NOT NULL, default `now()` | |
| `updated_at` | `timestamp` | NOT NULL, default `now()` | |

**Why store asset URLs directly on scenes instead of a separate `assets` table?**  
Each scene has a fixed, known set of assets: start image, end image, video, audio. There is no variable number of assets per scene in MVP. A separate assets table would add a join with no benefit. If multi-image or multi-take support is added later, an assets table can be introduced at that point.

**Why store both `stage` and `status`?**  
`stage` is *where* the scene is on the board (which column). `status` is *what's happening* within that stage (is a job running?). They are orthogonal — a scene can be in the `images` stage with `status: idle` (waiting to be triggered), `generating` (job running), or `done` (ready to promote).

**Why keep `error_message`?**  
Replicate and Trigger.dev jobs fail. Users need to know why (e.g. "model cold start timeout") so they can retry or adjust their prompt. Without this, the UI can only show a generic error.

## Indexes

These are the queries that will run on every page load. Index accordingly.

| Index | Table | Column(s) | Reason |
|---|---|---|---|
| `idx_projects_user_id` | `projects` | `user_id` | Dashboard query: fetch all projects for a user |
| `idx_scenes_project_id` | `scenes` | `project_id` | Workspace query: fetch all scenes for a project |
| `idx_scenes_project_stage` | `scenes` | `project_id, stage` | Kanban query: fetch scenes per column |

## Cascades & Deletion

| Action | Behaviour |
|---|---|
| Delete a `user` | Manual cleanup required (handled in a server function, not DB cascade) — delete R2 assets first, then DB rows |
| Delete a `project` | Cascades to `scenes` via `ON DELETE CASCADE` |
| Delete a `scene` | R2 assets are orphaned unless explicitly deleted before removing the row |

**Why no cascade on user delete?**  
R2 assets need to be cleaned up before the DB rows are removed. A DB cascade would delete the rows (and their R2 URLs) before we have a chance to purge the files. A server function handles this in the right order.

---

## What's Intentionally Not Here

| Thing | Why not |
|---|---|
| `api_keys` table | The Replicate key lives on the `users` row — one key per user, no need for a separate table in MVP |
| `jobs` table | Trigger.dev is the job store. We only keep the job ID on the scene row for reference |
| `assets` table | Fixed asset set per scene makes this unnecessary for MVP |
| `teams` / `organizations` | Out of scope for MVP |
| `audit_log` | Out of scope for MVP |
